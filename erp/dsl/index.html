Entity


Entity is something that will be stored in the database. 

entity Company

Here Company is the name of the entity.

It is concessional to start an entity name for a capital letter and user a singular form of the word. To user several words, use upper CamelCase:

  • Company - good
  • InvoiceItem - good
  • company - bad
  • Companies - bad
  • Invoice_Item - bad

Fields


entity Company {
  field code: Int
  field name: String
}

Two fields are declared here: code and name. The code is integer and name is string. Convensionally, lower CamelCase is used for field names:

  • name - good
  • itemQuantity - good
  • Name - bad
  • Companies - bad
  • item_Quantity - bad

You can set a default value for a field: field name: String = "Default Name". If you have a default value, you can omit the data type declaration: field name = "Default Name". This field will be considered String automatically because "Default Name" is String.

Compute Fields


Compute fields are not stored in the database but they are visible to the user. To declare a compute property, use the compute keyword instead of field and set an expression for calculation. 

entity Invoice {
  field qty: Dec[2]
  compute price: Dec[2] = total/qty
  field total: Dec[2]
}

Here we declare the compute field price. You can simplify the declaration and omit the datatype. It will be infered automatically: field price = total/qty.

This compute field will be read only. If you want to make it editable, yuo have to use the update modifier: 

compute price = total/qty {
  update {
    total = qty*price
  }
}

This means that when the price field is changed, we will update total with qty*price.

Reference Fields


A reference field contains reference to another or the same entity. To do this, you can easily use the name of the entity as data type after colon: 

entity Invoice {
  field company: Company
}

entity Company {
  field title: String
}

You can make references to any entities but you have to define a string representation for the entity because the reference field will use the string representation to show the reference to user. To do this, you need to declare a title field or function and define an expression how to convert the entity to string. 

entity Invoice {
  field number: Int
  field date: Date
  
  compute title: String = number + " date " + date
}

By default a selector will show only the title field. However you can set a custom layout. For this, you should declare ref view. 

entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
}

After this all references to the Company entity will show the code field under the title field. The same layout will be also used in a selector list. But you can customise this as well providing a refList view. 
entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
  view refList = table {title, code}
}

Now selector will show a table with two columns instead of list of the ref layout.

If you want to set a custom layout for one reference only, you can use view modifier for a reference field. 

entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
  view ref2 = row {title, code}
}  
entity Invoice {
  field company: Company
  view form = net {
    company => field(view = Company.ref2)
  }
}

Bidirectional references

When you have reference in one directions, sometimes you would like to easily find items which refer this item. To achive this you have to declare a backward reference field and using the by modifier link two reference fields together. Then the system will keep them in sync. 

entity Invoice {
  field company: Company
}
entity Company {
  field invoices: [Invoice] {
    by = company
  }
}

In the example, invoices have a company reference. In the Company entity, the invoices backward reference is declared.

Modifiers

by: I => Unit
Link two reference field together so that the system will keep them in sync.

Enum Fields


If you have a fixed set of variants to choose, it is recommended to use an enum field. 

enum TransactionType {
  case Invoice
  case Sale
}

entity Transaction {
  field type: TransactionType
}

TransactionType is an enum class, and you can see field type is an enum field. TransactionType contains to cases Invoice and Sale. This field will be displayed to use as a combo box and user will be able to select an option either Invoice or Sale. Conventionally, cases should use upper CamelCase, so they should be started from Capital letter.

You can use label modifier as usual to provide a text which user will see. 

enum TransactionType {
  case Invoice {
    label = "Customer Invoice"
  }
  case Sale {
    label = "We Sale"
  }
}

If you put your enum type in square brackets, user will be able to choose few options together. 
enum Tag {
  case Tag1
  case Tag2
  case Tag3
} 
entity Company {
  field tags: [Tag]
}

Collection Fields


Collection fields allow to add many items of similar type into an entity. 

entity Foo {
  field strings: [String]
  field companies: [Company]
}

In the example the strings field can contain many strings, the companies field is one to many reference field. It can contain many companies.

In a collection field you can also store a composite structures when you need to have few fields linked to each other. 

entity Invoice {
  field items: [InvoiceItem]
}

class InvoiceItem {
  field product: Product
  field qty: Dec[2]
}

InvoiceItem is a class. This means that it can be stored in the database only as a part of another entity. It does not have an id and cannot be selected itself, but only with another entity which will use it.

items is a collection field because the datatype declared in square brackets items. This means that there are many items. This field won't be visible in list view, and will be represented as a table in form by default.

Access


By default there is Admin access group declared. This means that the user will have full access to everything. In a workspace there must be at least one user with Admin permissions. If you want to give only limitted permissions to the user you should create an access group and assign it to the user. If user do not have access groups, he will have no permissions. 

access Accountant {
  entity Company = readWrite
  entity Product = readOnly
}

In the example, we created a new access group and granted a full access to the Company entity and a read only access to Product. 

access Accountant {
  entity Product {
    read = true
    write = type == Service
  }
}

In this example, we provided custom conditions. This means that a user will be able to read all products but save or delete only if product's type is Service. 

access Accountant {
  entity Product {
    read = type == Service
    write = old.isEmpty || new.isEmpty || new?.type == old?.type
  }
}

You can access to the old and new instances of the entity. It is optional instances. If user adds a new item, old will be empty. If user deletes an item, new will be empty. If user updates an item, both of them will be full.

In the example, user will be able to create and delete items and will be able to modify only if the type field is unmodified.

Modifiers

read: T => Bool
Defines a predicate for read access.
read: (T, old: T?, new: T?) => Bool
Defines a predicate for write access.

Introduction


Entities always contain two view. list is to display many items and form is to display one item. If these views are not defined in an entity, they will be generated automatically. You can use also different names for your views. 

entity Company {
  field code: Int
  field name: String
  
  view list = table{code, name}
}

| code | name     |
-------------------
| 1    | Company1 |
| 2    | Company2 |

To declare a view, you should start it with keyword view. Then you can use any name for view and after = you can define layout.

is a type of layout. In braces there sublayouts. In this case there are two fields code and name. They will be displayed as table columns. 

entity Invoice {
  field number: String
  field date: Date
  field items: [InvoiceItem]
  view form = column {
    net {
      number
      date
    }
    items => table(readOnly){product, qty}
  }
}
class InvoiceItem {
  field product: Product
  field qty: Dec[2]
}

number:| numberField  | 
  date:| dateField    |

| product  | qty |
------------------
| Product1 | 10  |
| Product2 | 23  |

This is more complex example. column puts sub-layouts one under another vertically. net create a layout with labels and fields as a form. Every field will have its label before it. All fields will be aligned.

items => means that table will show then items collection field with columns product and qty. After layout type in brackets there could be modifiers. In this example table(readOnly), readOnly is a modifier which makes the table always readOnly.

Button


You can use buttons to trigger some custom action when user press them. 

entity Company {
  field code: Int
  field name: String
  
  view form = column {
    net {
      code
      name
    }
    button("Update name & Code") {
      code = 1
      name = "Test"
    }
  }
}

button("Update name & Code") created a button with title "Update name & Code". This is a shorter construction to button(label = "Update name & Code"). So we simply set the label modifier for button. If you want to add some more modifiers you can write: button("Update name & Code", span = 2)

In braces there is an action script which will be executed when user presses the button. In this example, we set code field to 1 and name to "Test".

Styles


There two ways how to set colors, fonts and other styles to your component:

  • css modifier
  • style declaration and modifier

Both ways will use CSS. You can use the CSS reference to find out what options are available.

css modifier

Add CSS inline style to the component. 

view list = net {
  code
  name => field(css = "background: red; color: white")
}

style declaration and modifier

Reference to style if you want to use a common style form few components.

entity Company {
  field code: String
  field name: String
  view list = net {
    code => field(style = fieldStyle)
    name => field(style = fieldStyle)
  }
  style fieldStyle {
    "background: red;
     color: white"
  }
}

Here the fieldStyle style is declared. It is available also outside the entity by Companies.fieldStyle and can be declared in any places. For example there could be a class Styles which contains all styles. It is a good practice if you use the same styles in different entities.

"You can use selectors using & literal to refer the style name." 

style fieldStyle {
  "& > .field {
    color: red;
  }"
}

Language


Operators


=
Set operator changes value of the left operand to the value of right operand: price = total/qty
Dot .
Get item property: 
entity Product {
  field supplier: Company
  func supplierName: String = supplier.name
}
Optional dot ?.

If an item is optional and you want to get its property, you can use the optinal dot operator. This will return an optional value as well: 

entity Product {
  field supplier: Company?
  func optionalSupplierName: String? = supplier?.name
}

If the right operand is also optional, this operator will join two optional values and return optional instead of optional of optional. If the right operand is also collection, this operator will a collection instead of optional collection. 

class InvoiceItem {
  field product: Product?
  
  func productColor: Color? = product?.color
  func productSuppliers: [Company] = product?.suppliers
}
entity Product {
  field color: Color?
  field suppliers: [Company]
}
Option unwrap !
Unwrap option value. If option is empty, computation will fail: 
entity Product {
  field supplier: Company?
  func notOptionalSupplier: Company = supplier!
  func notOptionalSupplierName: String = supplier!.name
}
Collection dot *.

This operator takes a collection and return another collection of field values: 

entity Product {
  field suppliers: [Company]
  func supplierNames: [String] = suppliers*.name
}

If the right operand is also optional, this operator will return collection of only full fields instead of collection of options. If the right operand is also collection, this operator will join the collections and return one collection instead of collection of collections. 

class Invoice {
  field products: [Product]
  
  func productColors: [Color] = product*.color
  func productSuppliers: [Company] = product*.suppliers
}
entity Product {
  field color: Color?
  field suppliers: [Company]
}

Add two numeric values 1 + 2 == 3. The same operator can be used to concatenate two string. "1" + "2" == "12" If one of operands are not numbers both of them will be converted to string and joined: "1" + 2 == "12" Additionally, you can join two collection together or add an element to collection: [1, 2] + [2, 3] == [1, 2, 2, 3], [1, 2] + 3 == [1, 2, 3], 1 + [2, 3] == [1, 2, 3]
-

/

Greater operator. 2 > 1

Greater or equal operator. 2 >= 1

Logical operators

Inverse unary logical operator returns true if the operand is false and visa versa. !true == false, !false == true

Binary operators have a natural order and you can use brackets to control the order: 1 + 2*3 == 1 + (2*3) != (1 + 2)*3 If operators are in the same group the order will be from left to right.

  2. ||
  3. &&
  4. ==, !=, >, >=, <, <=
  5. +, -
  6. *, /

An entity will be stored in the database directly and will have an id. You can makes queries on this.

A universal class that can contain fields, function, etc as well as entity, but it can be stored in the database only as an embedded entity field or collection field. You cannot select them directly only through its parent entity.

Traits contain something common for few entities or classes. Classes and entities can implement many traits. See Inheritance. You can select all entities which implement the trait.
enum
Enum is a datatype which can have only few values. For example statuses.

Case for enum. Object related.

Inheritance


trait HasCode {
  field code: String
}
trait ActionItem {
  func name: String
  field priority: Int
  field status: Status
}

entity Product extends HasCode {
  field description: String
}
entity Company extends HasCode with ActionItem {
  field name: String
}

Traits can be a base for entities and classes. Entity can inherit multiple traits. All fields declared in the traits will be available in the entity. Product will have code field declared in the HasCode trait. It declares alse description field. So it will have these two fields.

In this example the Company entity will have code field from HasCode trait and priority and status from ActionItem. Field name is declared in the Company. In ActionItem name is declared as func. This means that the trait itself does not create a field but require form entity to provide either a field, compute field or a function with the same signature. This is more flexible and allows to use existing fields to calculate value and satisfy the trait's requirements.

It allows to avoid duplication and reuse the same structure in similar entities. It also allows to define a contract which entities should satisfy when implementing this trait.

Extensions


entity Company {
  field name: String
}
extend entity Company {
  field description: String
}

You can add field into existing entity or class using the extend entity construction. In this example, Company will have the description field as well as name. It allows to add user-specific fields into existing applications.

Syntax


Class ::=
[extend] {entity | class | trait | enum} <Name> [extends <SuperClass>*] {
  {<Field> | <ComputeField> | <Function> | <View> | <Case>}*
}

Field ::= field <Name>: [<DataType>] [= <Expression>] [{ <Modifier>* }]
ComputeField ::= compute <Name>: [<DataType>] = <Expression> [{ <Modifier>* }]
Function ::= func <Name>[(<Param>*)]: [<DataType>] = <Expression>
View ::= view <Name> = <Layout>
Case ::= case <Name> [{ <Modifier>* }]

Modifier ::= <ModifierName> {[= <Expression>] | [{ <Expression>* }] }

View ::= view <Name> = <Layout>

Layout ::= [<Expression> => ] [<LayoutType>[(<Modifier>*)] [{ <Layout>* }]]
