Entity


Entity is something that will be stored in the database. 

entity Company

Here Company is the name of the entity.

It is concessional to start an entity name for a capital letter and user a singular form of the word. To user several words, use upper CamelCase:

  • Company - good
  • InvoiceItem - good
  • company - bad
  • Companies - bad
  • Invoice_Item - bad

Fields


entity Company {
  field code: Int
  field name: String
}

Two fields are declared here: code and name. The code is integer and name is string. Convensionally, lower CamelCase is used for field names:

  • name - good
  • itemQuantity - good
  • Name - bad
  • Companies - bad
  • item_Quantity - bad

You can set a default value for a field: field name: String = "Default Name". If you have a default value, you can omit the data type declaration: field name = "Default Name". This field will be considered String automatically because "Default Name" is String.

Compute Fields


Compute fields are not stored in the database but they are visible to the user. To declare a compute property, use the compute keyword instead of field and set an expression for calculation. 

entity Invoice {
  field qty: Dec[2]
  compute price: Dec[2] = total/qty
  field total: Dec[2]
}

Here we declare the compute field price. You can simplify the declaration and omit the datatype. It will be infered automatically: field price = total/qty.

This compute field will be read only. If you want to make it editable, yuo have to use the update modifier: 

compute price = total/qty {
  update = {
    total = qty*price
  }
}

This means that when the price field is changed, we will update total with qty*price.

Modifiers

update: (T) => Unit
To make the compute field editable. See the example above.

Reference Fields


A reference field contains reference to another or the same entity. To do this, you can easily use the name of the entity as data type after colon: 

entity Invoice {
  field company: Company
}

entity Company {
  field title: String
}

You can make references to any entities but you have to define a string representation for the entity because the reference field will use the string representation to show the reference to user. To do this, you need to declare a title field or function and define an expression how to convert the entity to string. 

entity Invoice {
  field number: Int
  field date: Date
  
  compute title: String = number + " date " + date
}

By default a selector will show only the title field. However you can set a custom layout. For this, you should declare ref view. 

entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
}

After this all references to the Company entity will show the code field under the title field. The same layout will be also used in a selector list. But you can customise this as well providing a refList view. 
entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
  view refList = table {title, code}
}

Now selector will show a table with two columns instead of list of the ref layout.

If you want to set a custom layout for one reference only, you can use view modifier for a reference field. 

entity Company {
  field code: String
  field title: String
  view ref = column {
    title
    code
  }
  view ref2 = row {title, code}
}  
entity Invoice {
  field company: Company
  view form = grid {
    company >> field(view = Company.ref2)
  }
}

Bidirectional references

When you have reference in one directions, sometimes you would like to easily find items which refer this item. To achieve this you have to declare a backward reference field and using the by modifier link two reference fields together. Then the system will keep them in sync. 

entity Invoice {
  field company: Company
}
entity Company {
  field invoices: [Invoice] by company
}

In the example, invoices have a company reference. In the Company entity, the invoices backward reference is declared.

Modifiers

by: I => Unit
Link two reference field together so that the system will keep them in sync.

Reference field mode

References can work in three modes:

edit
This is a default mode and it means that user can select a reference.
read
This creates a read only reference field. A usual use case is backward reference like in the example: 
entity Invoice {
  field client: Company
}
entity Company {
  field invoices: [Invoice] read by client
}
In this case, a user will not be able an existing invoice and assign it to a company but he will be able to see all invoices for the company and will be able to add a new invoice.
inline
Inline references will look similar to an embedded field. It works well if you want to create a connection between entities with some attributes: 
entity Person {
  field title: String
  field roles: [OrganisationPersonRole] inline by person
}
entity Organisation {
  field title: String
  field employees: [OrganisationPersonRole] inline by organisation
}

entity OrganisationPersonRole {
  field person: Person 
  field organisation: Organisation 
  field role: Role 

  field startDate: Date
  field endDate: Date?
}

entity Role {
  field title: String
}

Enum Fields


If you have a fixed set of variants to choose, it is recommended to use an enum field. 

enum TransactionType {
  case Invoice
  case Sale
}

entity Transaction {
  field type: TransactionType
}

TransactionType is an enum class, and you can see field type is an enum field. TransactionType contains to cases Invoice and Sale. This field will be displayed to use as a combo box and user will be able to select an option either Invoice or Sale. Conventionally, cases should use upper CamelCase, so they should be started from Capital letter.

To provide a text which user will see you can write it as in the example or use label modifier as usual. 

enum TransactionType {
  case Invoice = "Customer Invoice"
  case Sale = "We Sale"
}

If you put your enum type in square brackets, user will be able to choose few options together. 
enum Tag {
  case Tag1
  case Tag2
  case Tag3
} 
entity Company {
  field tags: [Tag]
}

Enums can have parameters. You can use it as a table with predifined data. 

enum ClientCategory(discount: Dec[2]) {
  case Basic(discount = 0)
  case Premium(discount = 0.1)
}
entity Company {
  field category: ClientCategory
  func discount = category.discount
}

Optional Fields


If a field is optional, you have to mark it explicitly, otherwise the system will force user to fill the field in. 

entity Company {
  field title: String
  field notes: String?
}

The system will force user to fill the title field here but notes can be empty.

NOTE: Currently the system is not forcing users to fill the fields but it will in the future.

Collection Fields


Collection fields allow to add many items of similar type into an entity. 

entity Foo {
  field strings: [String]
  field companies: [Company]
}

In the example the strings field can contain many strings, the companies field is one to many reference field. It can contain many companies.

In a collection field you can also store a composite structures when you need to have few fields linked to each other. 

entity Company {
  field phones: [Phone]
}
class Phone {
  field countryCode: Int
  field number: String
}

Notice that you cannot use references inside classes. If you need this, you have to use entities instead and make the reference field inline.

You can also declare inner classes. Then you will be able to access the parent class fields. 

entity Company {
  field countryCode: Int
  field phones: [Phone]
  class Phone {
    field countryCode: Int = this[Company].countryCode
    field number: String
  }
}

In this example, we set the countryCode code default value to the same value as in the company but user will be able to change it.

Functions


Functions can take parameters. A function without parameters is almost the same as compute field but the result will not be shown to user by default and you cannot make it editable. 

entity Foo {
  func add(x: Int, y: Int): Int = x + y
}

Field Modifiers


label: String 
field name: String {
  label = "Company Name"
}

This allows to set a name which will be used on UI. By default the name of the field will be used for this. CamelCase will be translated to user-friendly string automatically. So this modifier is required only if you want to show user a different label than the field's name.

visible: T => Bool
field status: Status
field foo: String {
  visible = status == Bar
}

Dynamic visibility of fields. In the example the foo field will be visible only when status will be Bar.

views: [Layout]

You can set a list of views in which the field should be shown if views are autogenerated. 

field foo: String {
  views = [list, form]
}
field bar: String {
  views = [form]
}

Here the foo field will be shown in both list and form views, but bar will be shown only in form.

Roles


By default there is the Admin role declared. This means that the user will have full access to everything. In a workspace there must be at least one user with the Admin permissions. A role can grant access to the user and can contain a main view. If user do not have any roles, he will have no permissions. 

role Accountant {
  entity Company = readWrite
  entity Product = readOnly
}

In the example, we created a new access group and granted a full access to the Company entity and a read only access to Product. You can declare a main view for the role. If the view is omitted, it will be autogenerated with menu items to the entities, traits and reports available for this role. 

role Accountant {
  entity Company = readWrite
  entity Product = readOnly
  view = menu Entities {
    entity Company
    entity Product
  }
}

In the example the menu is create explicitly. You can declare any view here with embedded reports, action buttons, etc. 
access Accountant {
  entity Product {
    read = true
    write = type == Service
  }
}

In this example, we provided custom conditions. This means that a user will be able to read all products but save or delete only if product's type is Service. 

access Accountant {
  entity Product {
    read = type == Service
    write = old.isEmpty || new.isEmpty || new?.type == old?.type
  }
}

You can access to the old and new instances of the entity. It is optional instances. If user adds a new item, old will be empty. If user deletes an item, new will be empty. If user updates an item, both of them will be full.

In the example, user will be able to create and delete items and will be able to modify only if the type field is unmodified.

You can get current user and compare it in access groups. 

entity Document {
  field owner: User = Current
}
access OnlyOwners {
  entity Document {
    read = owner.isCurrent
    write = owner.isCurrent
  }
}

In the example the owner field will be assigned by default to the current user. The access group will grant access to the items where the owner is current user.

Modifiers

read: T => Bool
Defines a predicate for read access.
read: (T, old: T?, new: T?) => Bool
Defines a predicate for write access.

Introduction


Entities always contain two view. list is to display many items and form is to display one item. If these views are not defined in an entity, they will be generated automatically. You can use also different names for your views. 

entity Company {
  field code: Int
  field name: String
  
  view list = table{code, name}
}

| code | name     |
-------------------
| 1    | Company1 |
| 2    | Company2 |

To declare a view, you should start it with keyword view. Then you can use any name for view and after = you can define layout.

is a type of layout. In braces there sublayouts. In this case there are two fields code and name. They will be displayed as table columns. 

entity Invoice {
  field number: String
  field date: Date
  field items: [InvoiceItem] inline
  view form = column {
    grid {
      number
      date
    }
    items >> table(editMode = AlwaysRead){product, qty}
  }
}

entity InvoiceItem {
  field product: Product
  field qty: Dec[2]
}

  date:| dateField    |

| product  | qty |
------------------

| Product2 | 23  |

This is more complex example. column puts sub-layouts one under another vertically. grid create a layout with labels and fields as a form. Every field will have its label before it. All fields will be aligned.

items >> means that table will show then items collection field with columns product and qty. After layout type in brackets there could be modifiers. In this example table(editMode = AlwaysRead), editMode = AlwaysRead is a modifier which makes the table always read only.

You can declare a menu in a layout. The menu items can open list views for an entity, report or trait.

Button


You can use buttons to trigger some custom action when user press them. 

entity Company {
  field code: Int
  field name: String
  
  view form = column {
    grid {
      code
      name
    }
    button("Update name & Code") {
      code = 1
      name = "Test"
    }
  }
}

button("Update name & Code") created a button with title "Update name & Code". This is a shorter construction to button(label = "Update name & Code"). So we simply set the label modifier for button. If you want to add some more modifiers you can write: button("Update name & Code", span = 2)

In braces there is an action script which will be executed when user presses the button. In this example, we set code field to 1 and name to "Test".

Styles


There two ways how to set colors, fonts and other styles to your component:

  • css modifier
  • style declaration and modifier

Both ways will use CSS. You can use the CSS reference to find out what options are available.

css modifier

Add CSS inline style to the component. 

view list = grid {
  code
  name >> field(css = "background: red; color: white")
}

style declaration and modifier

Reference to style if you want to use a common style form few components.

entity Company {
  field code: String
  field name: String
  view list = grid {
    code >> field(style = fieldStyle)
    name >> field(style = fieldStyle)
  }
  style fieldStyle {
    "background: red;
     color: white"
  }
}

Here the fieldStyle style is declared. It is available also outside the entity by Companies.fieldStyle and can be declared in any places. For example there could be a class Styles which contains all styles. It is a good practice if you use the same styles in different entities.

"You can use selectors using & literal to refer the style name." 

style fieldStyle {
  "& > .field {
    color: red;
  }"
}

Add CSS inline style to the component.

Reference to style if you want to use a common style form few components.
span: Int
Allows to expand the field on more than one column in parent layout. Default value is 1.

Reports


Reports can show data by query. You have to provide a query for the report and you can provide a list view. List view can be auto-generated. The simplest report will look like this: 

entity Invoice {
  field number: String
  field data: Date
  field total: Dec[2]
}
report InvoiceReport {
  query = Invoice.all
}

Here the Invoice entity is declared for example. In InvoiceReport query is defined to select all invoices. You can declare a list view in the report the same way as with entity. 

report InvoiceReport {
  query = Invoice.all
  view list = table{number, date, total}
}

You can add the report to a menu. 

menu ReportMenu {
  report InvoiceReport
}

Language


Operators


=
Set operator changes value of the left operand to the value of right operand: price = total/qty
Dot .
Get item property: 
entity Product {
  field supplier: Company
  func supplierName: String = supplier.name
}
Optional dot ?.

If an item is optional and you want to get its property, you can use the optinal dot operator. This will return an optional value as well: 

entity Product {
  field supplier: Company?
  func optionalSupplierName: String? = supplier?.name
}

If the right operand is also optional, this operator will join two optional values and return optional instead of optional of optional. If the right operand is also collection, this operator will a collection instead of optional collection. 

class InvoiceItem {
  field product: Product?
  
  func productColor: Color? = product?.color
  func productSuppliers: [Company] = product?.suppliers
}
entity Product {
  field color: Color?
  field suppliers: [Company]
}
Option unwrap !
Unwrap option value. If option is empty, computation will fail: 
entity Product {
  field supplier: Company?
  func notOptionalSupplier: Company = supplier!
  func notOptionalSupplierName: String = supplier!.name
}
Collection dot *.

This operator takes a collection and return another collection of field values: 

entity Product {
  field suppliers: [Company]
  func supplierNames: [String] = suppliers*.name
}

If the right operand is also optional, this operator will return collection of only full fields instead of collection of options. If the right operand is also collection, this operator will join the collections and return one collection instead of collection of collections. 

class Invoice {
  field products: [Product]
  
  func productColors: [Color] = product*.color
  func productSuppliers: [Company] = product*.suppliers
}
entity Product {
  field color: Color?
  field suppliers: [Company]
}
Substring [ .. ]

This operator allow to get substring for a string. The index of the first element is 0. "Test"[1 .. 2] == "es"

It applies for a string and can have two arguments. These two arguments can be either an index or a substring to search: "Test string"["es" .. "r"] == "est str".

You can omit an expression from one of the side, then it will start from the beggining or end of the string: "Test string"[ .. "r"] == "Test str".

You can use round brackets instead of square in one or in both sides. Then the possition will be excluded from the range:

"Test string"["es" .. "r"] == "est str"
"Test string"("es" .. "r"] == "t str"
"Test string"["es" .. "r") == "est st"
"Test string"("es" .. "r") == "t st"

"Test"[1 .. 2] == "es"
"Test"(1 .. 2] == "s"
"Test"[1 .. 2) == "e"
"Test"(1 .. 2) == ""

Math operators

+
Add two numeric values 1 + 2 == 3. The same operator can be used to concatenate two string. "1" + "2" == "12" If one of operands are not numbers both of them will be converted to string and joined: "1" + 2 == "12" Additionally, you can join two collection together or add an element to collection: [1, 2] + [2, 3] == [1, 2, 2, 3], [1, 2] + 3 == [1, 2, 3], 1 + [2, 3] == [1, 2, 3]
-
1 - 2 == -1.
*
2*3 == 6.
/
6/2 == 3.

Comparison operators

==
Equal operator returns true if two values are equal. 1 == 1 == true
!=
Not equal operator returns true if two values are not equal. 1 != 1 == false
>
Greater operator. 2 > 1
>=
Greater or equal operator. 2 >= 1
<
Less operator. 1 < 2
<=
Less or equal operator. 1 <= 2

Logical operators

!
Inverse unary logical operator returns true if the operand is false and visa versa. !true == false, !false == true
&&
Logical and operator returns true only if both operands are true otherwise false. true && true == true, true && false == false
||
Logical or operator returns false only if both operands are false otherwise true. false || false == false, true || false == true

Order

Binary operators have a natural order and you can use brackets to control the order: 1 + 2*3 == 1 + (2*3) != (1 + 2)*3 If operators are in the same group the order will be from left to right.
  1. =
  2. ||
  3. &&

Data Types


Dec[precision]

Decimal datatype. You have to set precision in square brackets. The expression can be a constant like field foo: Dec[2] or an expression field foo: Dec[currency.precision].

Int

Integer datatype. It is similar to Dec[0].

Bool

Boolean datatype. Can be either true or false.

String

String datatype.

dropPrefix(prefix: String): String, dropPrefixes(prefixes: [String]): String
If the string starts with the prefix it drops it otherwise it returns unmodified string. The dropPrefixes function call dropPrefix in the cycle for every prefix. 
"www.google.com".dropPrefix("www.") == "google.com"
"google.com".dropPrefix("www.") == "google.com"
"https://www.google.com".dropPrefixes(["http://", "https://", "www.") == "google.com"
dropSuffix(suffix: String): String, dropSuffixes(suffixes: [String]): String
If the string ends with the suffix it drops it otherwise it returns unmodified string. The dropSuffixes function call dropSuffix in the cycle for every suffix. 
"www.google.com".dropSuffix(".com") == "www.google"

Text

Formatted text. Multiline string editor will be used for fields with this type. The same functions can be applied for text as for string.

Date, Time and DateTime

Three datatypes for date, time and date&time together.

now
Returns a current date/datetime/time: Date.now, DateTime.now, Time.now

Option T?

This time contains either one item or nothing. T is any class here: String?, Company?.

isEmpty: Bool
Returns whether the option contains an item or nothing.

Collection [T]

Collection data type of any items. T is any class here: [String], [Company].

first: T?, last: T?
Optional first or last element of the collection. If the collection is empty this will return the empty value. 
[2, 3, 1].first == 2
[2, 3, 1].last == 1
[].first.isEmpty
ui selected: T?
Function returns currently selected value on UI or None. This function can be used only on UI. See also selectable modifier.
sum: T
Returns sum of the elements of the collection.
contains(item: T): Bool
Returns true if the item contains in the sequence.
filter(predicate: T => Bool): [T]
Returns only items which satisfy the predicate.
filter(predicate: T => Bool): T?
Returns the first item which satisfy the predicate if exists.
exists(predicate: T => Bool): Bool
Returns true if there is an item which satisfies the predicate in the list.
map[R](f: T => R): [R]
Apply f on every item of the collection and returns collection of the results.
flatMap[R](f: T => [R]): [R]
Apply f on every item of the collection and joins results into a new collection.
groupBy[K](by: T => K): [K: [T]]
Apply by on every item. If result is the same for few items, it will join results together. The function returns a mep with the key which is the result of the by function, and value is joined items with the same key.
sort(lt: (T, T) => Bool): [T]
Sort the collection according to function. If lt is true, it means that the first item should precend the second one.

Map [K: V]

Contains correlations between key (K) and value (V): [Int: String], [Company: Dec[2]].

agg[I, V1 where V = [I]](f: ([I], key: K) => V1): [K : V1]
Aggregate values if they are collection. Usually it is used along with the group function.

It executes a sequence of expressions and returns the value of the latest one.

To declare an immutable variable. This variable will be visible inside braces.

Conditional expression. If condition is true it return a positive value otherwise negative.

Boolean constants.

null

Classes


You can declare classes of different types:

entity
An entity will be stored in the database directly and will have an id. You can makes queries on this.
trait
Traits contain something common for few entities. Entities can implement many traits. See Inheritance. You can select all entities which implement the trait.
class
A universal class that can contain fields, function, etc as well as entity, but it can be stored in the database only as an embedded entity field or collection field. You cannot select them directly only through its parent entity. You cannot have reference fields inside a class.
interface
Interface are similar to traits but it can be implemented by classes as well as entities and you cannot select data for interface.
report
Declares report.
enum
Enum is a datatype which can have only few values. For example statuses.
access
Declares an access group.
menu
Declares a menu.

In these classes you can declare:
field
Fields are something which will stored in database.
func
A function. Function can have parameters and return a value.
compute
Compute fields are not storing in the database by default but they are shown to user as a field.
class
Inner class. Instance and fields of the current class are available inside the inner class.
view
Declare a form or list view. Object related.
case
Case for enum. Object related.
case
Declare a style. Object related.
entity
Describe an entity access for access groups or entity reference in menu. Object related.
query
Defines a query.

With declarations you can use some modifiers:

native
This means that the function is implemented in native code and here this is only a desription for the function.
object
Make this function related to object not to instance. This means that you cannot use other instance related declaration in this function. But you can use this function without an instance. 
class Foo {
  field instanceField: Int
  
  func instanceFunc: Int =  instanceField
  
  object func objectFunc: Int = 22 
  object func objectFunc2: Int = instanceField //!!!Error. You cannot use instance declarations here
}

Foo.objectFunc
Foo.instanceFunc //!!!Error. You cannot call it without an instance

Inheritance


trait HasCode {
  field code: String
}
trait ActionItem {
  func name: String
  field priority: Int
  field status: Status
}

entity Product extends HasCode {
  field description: String
}
entity Company extends HasCode with ActionItem {
  field name: String
}

Traits can be a base for entities and classes. Entity can inherit multiple traits. All fields declared in the traits will be available in the entity. Product will have code field declared in the HasCode trait. It declares alse description field. So it will have these two fields.

In this example the Company entity will have code field from HasCode trait and priority and status from ActionItem. Field name is declared in the Company. In ActionItem name is declared as func. This means that the trait itself does not create a field but require form entity to provide either a field, compute field or a function with the same signature. This is more flexible and allows to use existing fields to calculate value and satisfy the trait's requirements.

It allows to avoid duplication and reuse the same structure in similar entities. It also allows to define a contract which entities should satisfy when implementing this trait.

Classes cannot implement traits and you have to use an interface instead.

Extensions


entity Company {
  field name: String
}
extend entity Company {
  field description: String
}

You can add field into existing entity or class using the extend entity construction. In this example, Company will have the description field as well as name. It allows to add user-specific fields into existing applications.

Syntax


Class ::=
[extend] {entity | trait | class | interface | enum} <Name> [extends <SuperClass>*] {
  {<Field> | <ComputeField> | <Function> | <View> | <Case>}*
}

Field ::= field <Name>: [<DataType>] [= <Expression>] [{ <Modifier>* }]
ComputeField ::= compute <Name>: [<DataType>] = <Expression> [{ <Modifier>* }]
Function ::= func <Name>[(<Param>*)]: [<DataType>] = <Expression>
View ::= view <Name> = <Layout>
Case ::= case <Name> [{ <Modifier>* }]

Modifier ::= <ModifierName> {[= <Expression>] | [{ <Expression>* }] }

View ::= view <Name> = <Layout>

Layout ::= [<Expression> >> ] [<LayoutType>[(<Modifier>*)] [{ <Layout>* }]]
